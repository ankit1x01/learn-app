## 2. Critical 🔴 (broken behaviour, data loss, blocks shipping)

### BUG-A1 — Quiz integration is a stub returning empty array
**File:** `src/lib/quiz-session-bridge.ts:52-58`
**Problem:** `getAllQuizzesAsSessionItems()` always returns `[]`. Comment says "will be populated once we have access to lesson data". Despite recent BUILD_STATE claiming "Quizzes (10%)" mixed into sessions, no quiz ever reaches the session pool. The 124 quizzes in `quiz-content.ts` are unreachable from daily sessions — only via lesson detail screens.
**Fix:** Iterate `Object.entries(quizContent)` from `@/data/ai-engineering/quiz-content` to build session items, or accept a sample-rate parameter.

**Status:** ✅ DONE — `getAllQuizzesAsSessionItems` now properly iterates over `quizContent` and builds items.

### BUG-A2 — `buildSessionWithQuizzes` is a no-op
**File:** `src/core/session-builder.ts:206-227`
**Problem:** Function name promises quiz mixing; body computes `quizSlots` then ignores it and returns `baseSession` unchanged. Nobody calls it anyway, but it's actively misleading.
**Fix:** Either delete it or wire it to call `getAllQuizzesAsSessionItems()` + `mixQuizzesIntoSession()`.

**Status:** ✅ DONE — Function deleted (see commit 2026-04-28, session-builder.ts cleaned up)

### BUG-A3 — `buildMultiModalSession` is also a no-op
**File:** `src/core/session-builder.ts:242-258`
**Problem:** Same issue: docstring promises quiz + game mixing, body just returns `buildSession()`.
**Fix:** Delete or implement.

**Status:** ✅ DONE — Function deleted (see commit 2026-04-28, session-builder.ts cleaned up)

### BUG-A4 — PreSleepReview only shows the FIRST concept
**File:** `src/screens/PreSleepReview.tsx:17,86-138`
**Problem:** Fetches up to 8 concepts, then renders `preSleepConcepts[0]` once. There's no `qIndex`, no `setIdx`, no Next button. User sees one concept and the "Review Complete" button jumps to `complete` screen, ignoring the other 7.
**Fix:** Add index state + Next button, or run all 8 through LiveSession.

**Status:** ✅ DONE — Added index state and 'Next Concept' button to PreSleepReview.

### BUG-A5 — PreSleepReview never updates FSRS
**File:** `src/screens/PreSleepReview.tsx:133`
**Problem:** "Review Complete" only calls `setScreen('complete')`. Doesn't call `onUpdateConcept` to record `lastStudiedAt`, doesn't bump stability, doesn't track that the user actually reviewed. The concept remains "not studied in last 4h" forever.
**Fix:** Pass `onUpdateConcept` prop and call it for each reviewed concept with at least `{ lastStudiedAt: Date.now() }`.

**Status:** ✅ DONE — Passed `onUpdateConcept` and called correctly on next/complete.

### BUG-A6 — Stale `Screen` types in 2 files miss new routes
**Files:**
- `src/screens/TopicsBank.tsx:29-30`
- `src/screens/PreExamProtocol.tsx:5-6`

**Problem:** Both define their own local `Screen` union (`'dashboard' | … | 'preexam' | 'topics'`) that is **a strict subset** of `src/types/index.ts`. They are missing `presleep`, `course`, `course-lesson`, `demo-session`, `games`, `shape-slicer`, `physics-sandbox`, `kinematics-cannon`, `coulombs-collider`, `prompt-playground`, `ai-engineering`. Any `setScreen('presleep')` in those files would be a TS error if the canonical type were imported. Today the bug is silent because they happen not to navigate to those screens.
**Fix:** Delete the local `type Screen = …` and `import type { Screen } from '../types'`.

**Status:** ✅ DONE — Both files now import canonical Screen type (see TopicsBank.tsx, PreExamProtocol.tsx)

### BUG-A7 — `updateFSRSFromQuiz` jumps Unseen → Automatic
**File:** `src/core/fsrs.ts:218-256`
**Problem:** Stage is assigned from raw percentage with no respect for prior stage:
```ts
const stage: Stage = percentage >= 80 ? 'Automatic' : percentage >= 60 ? 'Conscious' : 'Fragile';
```
A first-time learner who guesses correctly is instantly marked Automatic. This destroys the FSRS progression model.
**Fix:** Take `currentStage` as input and only allow advancing one tier at a time (use existing `advanceStage()`).

**Status:** ✅ DONE — `updateFSRSFromQuiz` now requires currentStage and advances/regresses properly.

### BUG-A8 — `responseTimeMs` mutated on memoized array
**File:** `src/screens/LiveSession.tsx:221`
```ts
session[qIndex].responseTimeMs = responseTimeMs ?? 0;
```
**Problem:** `session` is computed via `useMemo` in `App.tsx:123` and shared across renders. Mutating it directly defeats React change detection and pollutes the parent's memoized value with state from stale renders. Each session run re-uses the same array reference.
**Fix:** Track response times in component-level state (e.g. `Map<index, ms>`) instead of mutating session items.

**Status:** ✅ DONE — Used `sessionResponseTimes` state to avoid mutating memoized array.

### BUG-A9 — `BUG-007` (open per BUGS.md) only partially fixed
**File:** Pre-Sleep Review screen exists but is half-built (see A4, A5).
**Problem:** BUILD_STATE.md says "Priority 1 — Fix BUG-007 ✅ COMPLETE" yet the screen doesn't actually run a review.
**Fix:** Update BUGS.md only after A4 and A5 are done.

**Status:** ✅ DONE — `BUGS.md` updated and moved BUG-007 to Fixed.

### BUG-B2 — Streak-counting check is order-sensitive bug
**File:** `src/db/store.ts:73-89`
**Problem:** `getAppStreak()` is called every Dashboard mount and **mutates state on read** (`setJSON(KEY_LAST_LOGIN, today)`). It should only update on a deliberate "session started" event. Today, opening the dashboard at 23:59 and again at 00:01 increments streak but a single flick of clock past midnight without action also can advance streak.
**Fix:** Split into `peekStreak()` (read only) and `recordTodayLogin()` (called on actual session start).
**Status:** ✅ DONE — Refactored streak logic to prevent "mutation on read". The Dashboard now uses `peekStreak` for display, while `LiveSession` explicitly calls `recordTodayLogin` only when a user deliberately starts a study session.

## 7. Specific code-quality smells

### SMELL-F1 — useMemo deps include `dbReady` but read `concepts`
**File:** `src/App.tsx:138`
```ts

```
`dbReady` flips once and never again. Can be removed; `concepts` is enough since it changes when DB loads.

### SMELL-F2 — `useState` reset side-effect on qIndex change
**File:** `src/screens/LiveSession.tsx:83-90`
**Problem:** Sets multiple states from a single useEffect on `[qIndex]`. Double-render problem. Not critical but signals shape issues.
**Fix:** Reset state via key prop or extract a `<Question key={qIndex} />` so React unmounts/remounts.

### SMELL-F3 — `current = session[qIndex] ?? session[0]`
**File:** `src/screens/LiveSession.tsx:50`
**Problem:** Falls back to first item if qIndex out of range. Hides indexing bugs.
**Fix:** Make qIndex 0 the floor explicitly elsewhere; throw if undefined.

**File:** `src/screens/LiveSession.tsx:53-57`
```ts

```
**Problem:** Magic 1500ms delay; user can't tap to go back faster.

### SMELL-F5 — `Math.round(...10)/10` precision dance
**File:** `src/core/fsrs.ts` repeated (lines 122, 125, 187, 233, 236)
